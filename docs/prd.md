## 1. 제품 개요 (Product Overview)

### 1.1 배경 및 목적

2026년 2월 26일 한국회계기준원(KSSB)이 KSDS(한국 지속가능성공시기준)를 확정 발표하면서, 코스피 상장사는 FY2028부터 단계적으로 ESG 의무 공시를 제출해야 한다. 기존 esg-t1은 단일 법인·Scope 1/2 중심 MVP를 목표로 설계되었으나, 실제 기업 환경(다법인 연결 구조, Scope 3 공급망 배출, 외부 검증 의무화)을 반영하지 못했다.

esg-t2는 esg-t1에서 검증된 기술 스택과 아키텍처 원칙을 계승하면서, 아래 세 가지를 MVP 범위로 격상한다:

1. **다법인 연결 공시** — 연결 자산총액 기준이 규제 적용 임계값이므로, 단일 법인 시스템은 상용화 불가
2. **Scope 3 배출량 (Category 1·2·11)** — GHG Protocol Phase 1 Update(2026-03)로 95% 포함 임계값 신설; 데이터 품질 투명성 요구 급상승
3. **외부 검증 워크스페이스(VW)** — 공시 의무화와 함께 감사인의 시스템 직접 접근 요구 증가

### 1.2 비전

> "코스피 상장사가 KSSB 의무 공시 시점(FY2028)에 맞춰 지체 없이 ESG 데이터를 수집·계산·검증·제출할 수 있도록 지원하는 신뢰할 수 있는 플랫폼"

### 1.3 esg-t1 대비 주요 변경

| 항목 | esg-t1 | esg-t2 |
|---|---|---|
| 규제 기준 | ISSB S2 + K-ESG 핵심 항목 | **KSSB 1/2** + ISSB S1/S2 수렴 |
| 법인 범위 | 단일 법인 | **다법인 연결 최소 3개** |
| GHG Scope | Scope 1+2 | **Scope 1+2+3 (Cat. 1·2·11)** |
| 외부 검증 | M+1 (미래) | **MVP 포함** |
| 이중 중대성 | 장기(L) | M+1 |
| GRI/SASB | 장기(L) | M+1 |
| iXBRL | 미정 | MVP 데이터 모델 설계, M+1 렌더링 |
| AI 보조 | 장기(L) | M+1 |

## 2. 용어 정의 (Glossary)

| 용어 | 정의 |
|---|---|
| Scope 1 / 2 / 3 | 온실가스 배출 분류. 1=직접 배출, 2=구매 전력·열 간접 배출, 3=가치사슬 상하류 기타 간접 배출(15+1개 카테고리) |
| Location-based / Market-based | Scope 2 산출 방식. 전자는 전력망 평균 계수, 후자는 계약 기반 계수(재생에너지 인증서 포함) |
| GWP (Global Warming Potential) | 온실가스별 지구온난화 기여도 환산계수. IPCC AR5·AR6 버전 구분 |
| GHG Protocol | 글로벌 표준 온실가스 산정·보고 지침 |
| KSSB 1 / KSSB 2 | 한국지속가능성공시기준. 1=일반 지속가능성(IFRS S1 수렴), 2=기후 관련(IFRS S2 수렴). 2026-02-26 확정 |
| IFRS S1 / S2 | 국제지속가능성공시기준. S1=일반, S2=기후. KSSB 1/2의 국제 기반 |
| ISSB | 국제지속가능성기준위원회. IFRS S1/S2 발행 기관 |
| TCFD | 기후 관련 재무정보공시 태스크포스. KSSB 2에 내재 |
| GRI / SASB / CSRD / ESRS / CDP | 주요 글로벌 공시·평가 프레임워크 |
| 이중 중대성 (DMA) | 재무적 중요성과 영향 중요성을 함께 평가. CSRD/ESRS 핵심 원칙 |
| Facilitated Emissions | GHG Protocol Category 16. 기업이 제3자 활동을 가능하게 하는 배출 |
| Audit Trail | 데이터 변경 이력(Who/When/What/Why) 로그 |
| Data Lineage | 원천 → 정제 → 산식 → 보고서까지의 데이터 흐름 추적 |
| Assurance | 제3자 검증. 제한적/합리적 보증 구분 (ISAE 3000/3410) |
| Evidence (증빙) | 데이터 포인트를 뒷받침하는 원시 문서(PDF/이미지/엑셀) |
| Snapshot | 특정 시점 데이터를 불변으로 고정한 복사본. 공시·검증·재공시용 |
| RBAC / ABAC | 역할 기반 / 속성 기반 접근 제어 |
| iXBRL / XBRL | 공시용 기계 판독 가능 데이터 포맷 |
| SBTi | 과학기반 감축 목표 이니셔티브 |
| RTO / RPO | 복구 목표 시간 / 복구 목표 시점 |
| Tenant | 다중테넌트 환경에서 독립된 기업 단위. 법인(LegalEntity)의 상위 개념 |

## 4. 규제 요건 (Regulatory Requirements)

> 상세 내용은 `docs/regulatory.md` 참조. 아래는 시스템 설계에 직접 영향을 주는 요건만 발췌.

### 4.1 KSSB 1/2 필수 공시 항목 (MVP 범위)

**KSSB 2 (기후 관련) — 의무**

| 영역 | 필수 공시 내용 | esg-t2 구현 모듈 |
|---|---|---|
| 지배구조 | 기후 관련 위험·기회 감독 구조 | `gov` 모듈 (M+1) |
| 전략 | 기후 관련 위험·기회가 비즈니스에 미치는 영향 | `rpt` 보고서 섹션 |
| 위험관리 | 기후 위험 식별·평가·관리 프로세스 | `dma` 모듈 (M+1) |
| 지표·목표 | **Scope 1·2·3 GHG 배출량**, 기후 관련 목표 | `ghg` 모듈 — MVP 핵심 |

**KSSB 1 (일반 지속가능성) — 자발적 권고 (M+1 지원)**

### 4.2 GHG Protocol 대응

| 요건 | 시스템 대응 |
|---|---|
| Scope 3 95% 포함 임계값 | `scope3_coverage_report` 자동 생성 (MVP) |
| Category 16 신설 | 데이터 모델 사전 설계; 계산 엔진은 2027년 기준 확정 후 |
| 지출 기반 → 공급업체별 데이터 투명성 | 데이터 품질 지표 (`data_quality_score`) 필드 추가 |

### 4.3 의무화 일정 (disclosure_schedule 메타테이블 관리)

| 기업 구분 | 최초 의무 공시 연도 | Scope 3 의무 시작 |
|---|---|---|
| 연결자산 30조+ | FY2028 (2029년 제출) | FY2031 또는 법안 통과 시 FY2029 |
| 연결자산 10조+ | FY2029 (2030년 제출) | FY2032 |
| 코스피 전체 | FY2030 (2031년 제출) | FY2033 |

> ⚠️ 이 일정은 `disclosure_schedule` 테이블에서 관리. 시스템 하드코딩 금지.

| 다국어 포털 UI | **M+1** | MVP는 한국어만 |
| 공급업체별 자동화 데이터 검증 룰 | **M+1** | MVP는 수동 검토·승인으로 처리 |

> **SUPPLIER 역할 제약**: 자사 데이터(`supplier_id` 일치)만 입력·조회 가능. RLS + 서비스 레이어 이중 방어.

## 5. 비기능 요구사항 (NFRs)

### 5.1 성능

| 지표 | 목표 |
|---|---|
| 페이지 응답시간 (P95) | ≤ 500ms (전체), ≤ 1,500ms (보고서 생성) |

| 동시 사용자 | 100명 동시 접속 안정 운영 |

### 5.2 가용성 및 복구

| 지표 | 목표 |
|---|---|
| 가용성 | 99.5% (월간) |
| RTO (복구 목표 시간) | ≤ 4시간 |
| RPO (복구 목표 시점) | ≤ 1시간 (일 4회 스냅샷) |

### 5.3 보안

| 요건 | 구현 |
|---|---|
| 인증 | JWT (Access 15분 / Refresh 7일) + Spring Security |
| 다법인 데이터 격리 | Row-Level Security (`tenant_id` + `entity_id` 복합 필터) |
| 암호화 | TLS 1.3 전송, AES-256 민감 필드 저장 |

- Spring Boot 4.0 내장 OpenTelemetry (OTLP exporter)
- Prometheus + Grafana 메트릭 대시보드

- AuditLog Hash Chain 무결성 자동 검증 스케줄러 (1일 1회)

## 11-A. 설계 원칙 (Design Principles)

1. **정책값 분리**: 공시 일정·배출계수·산식·프레임워크 매핑은 코드에 하드코딩하지 않는다. `disclosure_schedule`, `emission_factors`, formula YAML로 관리.
2. **불변성 (Immutability)**: 공시에 사용된 산출 결과·AuditLog·Snapshot은 INSERT-only. DB 권한 레벨에서 UPDATE/DELETE 박탈.
3. **재현성**: `[활동 데이터 버전 + 산식 버전 + 배출계수 버전]` 조합이 같으면 항상 동일 결과.
4. **증빙-산식-공시 연결 추적 (Evidence-to-Disclosure Traceability)**: 공시 수치 → 계산 결과 → 활동 데이터 → 증빙 파일까지 역추적 가능.
5. **감사 우선**: 모든 데이터 변경은 @Auditable AOP로 누락 없이 기록. AuditLog Hash Chain으로 위변조 탐지.
6. **보고서와 공시 분리**: 보고서 생성 후 반드시 검토·승인을 거쳐야 공시 가능. 미승인 데이터는 공시 산출물에 포함 불가.
7. **검증인 격리**: VERIFIER 역할은 지정 Snapshot 외 데이터 접근 불가. RLS + 도메인 레벨 이중 방어.
8. **확장 가능한 최소 구조**: MVP 코드가 M+1 추가 시 폐기되지 않도록 Spring Modulith 모듈 경계 설계.
