## 2. 스키마 설계 불일치·문제

### 2.1 tenants 테이블 컬럼 불일치

- [ ] **spec.md** `tenants` 테이블에 `code` 컬럼 없음 / **구현 계획 V1** `code VARCHAR(50) NOT NULL UNIQUE` 있음  
  → `code`가 필요한지 결정 후 spec.md 또는 구현 계획 중 하나를 수정.

- [ ] **spec.md** `tenants` 테이블에 `country_code` 없음 / **구현 계획 V1** `country_code CHAR(2) NOT NULL` 있음  
  → 테넌트(기업 단위)에 국가코드가 필요한지 검토. 법인(legal_entities)에 국가코드가 있으므로 중복일 수 있음.

### 2.2 legal_entities 테이블 설계 문제

- [ ] **spec.md** `legal_entities`에 `reporting_year INT NOT NULL` 컬럼 존재  
  → 법인은 다년도에 걸쳐 존재하는 마스터 데이터인데 특정 연도를 가지는 것이 설계 이상. `entity_type` 컬럼도 `entity_relationships`의 `method`와 중복 가능. 이 컬럼이 실제로 필요한지 검토.

- [ ] **spec.md** vs **구현 계획**: spec에는 `entity_type VARCHAR(30)` (consolidation_method), 구현 계획에는 `entity_type VARCHAR(50)` (PARENT/SUBSIDIARY/ASSOCIATE) + 별도 `method VARCHAR(50)` (EQUITY/OPERATIONAL_CONTROL)  
  → 통합 또는 분리 여부 결정 후 스키마 일치시킬 것.

### 2.3 emission_factors 테이블의 `year` 컬럼명

- [ ] **spec.md** `emission_factors`에 `year INT NOT NULL` — SQL 예약어 주의 규칙(CLAUDE.md, 01-domain-architecture.md)에 따르면 `reporting_year`로 바꿔야 함.  
  → `year` → `reporting_year`로 변경. H2 테스트에서 파싱 오류 유발 가능.

### 2.4 마이그레이션 파일 번호 체계 불일치

- [ ] **plan.md Phase 0** = `V1__initial_schema.sql (tenants, legal_entities, disclosure_schedules)` — legal_entities가 V1에 포함  
  **task.md** = T-1-01 `V2__entity_tables.sql`, T-2-01 `V4__audit_tables.sql`  
  **구현 계획** = V1(tenants+disclosure_schedules), V2(seed data), V3(entity+auth), V5(audit)  
  → 세 문서에서 마이그레이션 파일 번호가 모두 다름. 최종 번호 체계를 하나로 결정하고 전 문서 동기화.

### 2.5 verification_snapshots 업데이트 방지 방법

- [ ] **spec.md** `CREATE RULE no_update_snapshot AS ON UPDATE TO verification_snapshots DO INSTEAD NOTHING;`  
  → PostgreSQL `RULE`은 레거시 기능으로 `INSTEAD NOTHING`은 오류를 던지지 않아 업데이트 시 조용히 무시됨 (데이터 보호 인식 불가). **DB 권한 박탈(`REVOKE UPDATE, DELETE ON verification_snapshots FROM app_user`)** + 트리거 방식으로 교체 검토.

### 2.6 audit_logs.tenant_id NOT NULL 여부

- [ ] **spec.md** `audit_logs`에 `tenant_id UUID NOT NULL` — 시스템 레벨 이벤트(테넌트 생성, 스케줄러 실행 등)는 tenant_id가 없음  
  → `NOT NULL` → `NULL 허용`으로 변경하거나 시스템 테넌트 UUID를 예약할지 결정.

### 2.7 disclosure_reports.content JSONB 저장 방식

- [ ] **spec.md** `disclosure_reports`의 `content JSONB NOT NULL` — 보고서 전체를 단일 JSONB 컬럼에 저장하면 변경 이력·필드별 조회·부분 업데이트가 어려움  
  → JSONB를 유지할지, 섹션별 별도 테이블(disclosure_sections)로 분리할지 결정.

## 7. .claude/rules 파일 기술적 오류

### 7.1 Outbox Poller의 `zone` 속성 오용

- [ ] **09-scheduler.md** Outbox Poller 예제: `@Scheduled(fixedDelay = 5000, zone = "Asia/Seoul")`  
  → `zone` 속성은 `cron` 표현식에만 적용됨. `fixedDelay`/`fixedRate`에는 무시됨 (Spring 문서 확인).  
  → `zone = "Asia/Seoul"` 제거. fixedDelay는 KST 개념 불필요. 해당 설명 오해 없도록 주석 수정.

### 7.2 `AuditAspect` 패키지 경로 vs 접근 가능성

- [ ] **11-modulith-events.md** `@Auditable`은 `shared/` 패키지에 있고, `AuditAspect`는 `audit/internal/`에 있음  
  → AuditAspect가 다른 모듈의 메서드(예: ghg 모듈 서비스)에 `@Auditable`을 감지하려면 `audit` 모듈이 다른 모듈 클래스를 scan해야 함. Spring Modulith에서 AOP cross-module 동작 가능 여부 확인.

## 12. 전반적 미결 설계 사항

### 12.1 RLS 정책의 마이그레이션 위치 결정

- [ ] **spec.md** `legal_entities` 테이블에 RLS 정책 DDL이 공통 마이그레이션(`db/migration`)에 포함되어 있음  
  → **08-persistence.md** + **규칙**: PostgreSQL 전용 DDL은 `db/migration-pg`에 배치  
  → `ALTER TABLE ... ENABLE ROW LEVEL SECURITY`, `CREATE POLICY ...`는 `db/migration-pg`로 이동. spec.md 스키마 예시도 위치 명시 필요.

### 12.2 Tenant 컨텍스트 전파 방법 미정

- [ ] **spec.md 보안 모델** RLS에서 `current_setting('app.current_tenant_id')` 사용 명시되어 있으나, 이 값을 `SET LOCAL`로 설정하는 필터·인터셉터 구현이 spec.md, task.md 어디에도 없음  
  → Phase 1 보안 설계에 `TenantContextInterceptor` (매 요청마다 `SET LOCAL app.current_tenant_id = ?`) 태스크 추가.

### 12.3 멀티 테넌트에서 VERIFIER 격리 RLS 설계 미완성

- [ ] **spec.md / 03-security.md** VERIFIER 전용 RLS: `WHERE snapshot_id = current_setting('app.verifier_snapshot_id')` 언급  
  → 이 세션 변수를 어떻게 설정하는지(로그인 시 지정? JWT claim?), tenant_isolation 정책과 어떻게 공존하는지 설계 내용 없음  
  → Phase 8 VW 설계 전에 VERIFIER RLS 작동 방식을 spec.md에 구체화.

### 12.4 Hash Chain 성능 설계 미정

- [ ] **spec.md / 05-async-concurrency.md** Hash Chain에 PESSIMISTIC_WRITE 락 사용 명시  
  → 대용량 AuditLog 시 락 경합이 bottleneck이 될 수 있음. DB Outbox를 통한 순차 처리로 이를 완화하는지, 또는 파티션별 Hash Chain 분리를 고려하는지 아키텍처 결정 문서화 필요.
