## 1. 전반적 평가

Gemini 리뷰는 **사실 오류가 없고** 아키텍처를 정확하게 이해한 상태에서 작성되었다.  
ChatGPT 리뷰(2026-05-18)와 달리 잘못된 전제나 이미 구현된 사항을 누락으로 오인하는 케이스가 없다.  
제안 4건 + 우선순위 재정렬 1건을 검토한다.

**긍정 평가 섹션 (별도 조치 불필요)**:
- 섹션 2: 아키텍처·규제 대응·TDD 프로세스 평가 — 모두 정확, 조치 없음
- 섹션 3 장점 4건 — 정확한 파악, 조치 없음
- 섹션 3 위험 #1(기술적 복잡도), #2(성능 병목) — 인지하고 있는 위험. 성능 병목은 A-5(Capacity Planning)으로 이미 반영

### G-1: UI/UX 프로토타이핑 조기 도입

**Gemini 제안**  
Phase 9까지 기다리지 말고 Phase 3(GHG 계산) 단계부터 핵심 화면(데이터 입력 폼, 대시보드)의 와이어프레임을 확정하여 백엔드 API 설계와 동기화.

**현재 상태 대조**
- `CLAUDE.md`: "UI 개발 게이트: UI Phase 시작 전 반드시 사용자 승인 후 착수" — Phase 9~11이 UI Phase
- `prd.md §5.6` UJ-01, UJ-02 User Journey 정의로 주요 API 흐름 기준 확보
- `spec.md §5`: 주요 엔드포인트 목록 사전 정의
- `plan.md`: Phase별 OpenAPI 어노테이션 작성 체크리스트 포함

**판단: ⚠️ 조건부 적용**

| 항목 | 판단 |
|---|---|
| 별도 와이어프레임 작업 추가 | ❌ 미적용 — YAGNI 원칙; 와이어프레임은 별도 디자인 도구 필요, MVP 단계 오버헤드 |
| API 응답 DTO 구조 사전 합의 절차 | ✅ 적용 권장 — 백엔드 Phase 착수 전 핵심 API 응답 DTO를 spec.md에 먼저 작성 후 시작 |

**조치**: `plan.md` 각 백엔드 Phase 착수 체크리스트에 "핵심 API 응답 DTO 구조 spec.md 추가 확인 후 착수" 항목 추가를 권장. **즉시 파일 수정은 하지 않음** — plan.md 체크리스트는 실행 중에 현행화.

**이유**: 와이어프레임은 디자이너 리소스 없이 개발자가 작성하면 오히려 잘못된 가이드로 작용할 수 있음. 대신 OpenAPI 어노테이션 기반 API-first 설계(현재 plan.md에 포함)로 동일한 효과를 얻을 수 있음.

### G-2: 데이터 마이그레이션 전략 구체화

**Gemini 제안**  
기존 엑셀·타 시스템의 과거 데이터를 이관하는 `Data Migration Tool` 또는 `Batch Job` 상세 스펙 추가.

**현재 상태 대조**
- `prd.md F-03-02`: CSV/Excel 일괄 업로드 (신규 데이터 입력용) — 이관 전용 도구와 다름
- `prd.md M+1 목록`: 과거 데이터 이관 도구 없음 — **실제 누락**
- KSSB 2: 비교연도(전년도) 데이터 필수 → 최소 1년치 과거 데이터 이관 없이 첫 공시 불가능

**판단: ✅ 적용**

**이유**: 
1. KSSB 2 §33: 비교 공시 의무 — 전년도 실적 없이 보고서 생성 불가
2. F-03-02 CSV 업로드는 신규 데이터 입력 흐름, 과거 2~3년치 이관은 별도 배치 도구가 필요
3. 실제 기업 도입 시 "지금까지 엑셀로 관리한 데이터를 어떻게 넣나"가 1번 질문

**조치**: `prd.md` M+1 목록에 **M1-08: 과거 데이터 이관 도구** 추가

```
| M1-08 | 과거 데이터 이관 도구 | 기존 엑셀·ERP·타 ESG 시스템에서 과거 3개년 활동 데이터 일괄 이관; 검증·매핑·결과 보고서 포함 |
```

### G-3: Hash Chain 무결성 오류 시 복원 절차 (DR 강화)

**Gemini 제안**  
Hash Chain 무결성 오류 발생 시 구체적인 데이터 복원 절차(Recovery Playbook)를 ADR 수준에서 정리.

**현재 상태 대조**
- `prd.md F-09-03`: "일 4회 스냅샷; 분기 1회 복구 훈련; RTO ≤ 4시간" — 일반 DR 커버
- `spec.md §4.4`: 무결성 검증 스케줄러 코드 있으나, **오류 발생 시 대응 절차 없음**
- `docs/adr/`: ADR-006 (RLS), ADR-007 (Append-only) 있으나 Hash Chain 장애 대응 없음

**판단: ⚠️ 조건부 적용**

| 항목 | 판단 |
|---|---|
| 별도 ADR 작성 | ❌ 미적용 — ADR은 설계 결정 기록용; 운영 절차는 ADR의 범위가 아님 |
| spec.md audit 섹션에 오류 대응 절차 추가 | ✅ 적용 — 운영 가이드라인으로 spec.md §4.4에 서브섹션 추가 |

**이유**:
- Hash Chain 오류는 두 가지 원인: (1) DB 직접 조작에 의한 위변조, (2) 버그로 인한 직렬화 불일치
- 각각 대응이 다름. 위변조는 감사 기관 신고 절차, 버그는 Canonical JSON 재계산
- 이 절차가 spec.md에 있어야 운영팀이 장애 시 참조 가능

**조치**: `spec.md §4.4 audit 모듈` 끝에 "Hash Chain 오류 대응 운영 절차" 서브섹션 추가

### G-4: AI 이상값 탐지 조기 도입 (Phase 6 포함)

**Gemini 제안**  
M+1로 미뤄진 'AI 데이터 검증 보조' 기능을 Phase 6(데이터 수집)의 일부로 포함시켜 이상값 탐지를 실시간 지원.

**현재 상태 대조**
- `prd.md F-03-08`: **"누락·음수·전년 대비 50% 이상 급변 자동 경고"** — 통계적 이상값 탐지 이미 **MVP 포함**
- `prd.md M1-05`: "AI 데이터 검증 보조 — 이상값 탐지, 자동 완성 제안" — LLM 기반 기능 M+1 예정

**판단: ❌ 부적용**

**이유**:
1. Gemini가 F-03-08 존재를 인지하지 못한 것으로 보임. 규칙 기반 이상값 탐지는 이미 MVP에 포함
2. LLM 기반 AI 검증은 F-03-08과 다른 레이어:
   - F-03-08: 결정론적 룰(50% 급변, 음수, 누락) → Phase 6 구현 가능
   - M1-05: LLM 추론 기반(비정형 패턴 탐지, 자동완성) → 외부 API 의존성, 비용, 레이턴시 설계 필요
3. Phase 6에 LLM 통합 추가 시 scope creep 및 개발 기간 연장

**조치**: 없음. F-03-08이 Gemini가 요청하는 핵심 기능을 이미 커버함을 확인.

### G-5: Phase 우선순위 재정렬 (Phase 2 → 3 → 1 → 5 → 8)

**Gemini 제안**  
현재 Phase 1(테넌트·RLS) → Phase 2(AuditLog) 순서를 Phase 2 → Phase 3 → Phase 1 순으로 재정렬.  
"AuditLog가 시스템의 신뢰성 인프라이므로 Phase 1보다 먼저"가 근거.

**현재 상태 대조**  
`plan.md` Phase 순서:
```
Phase 0: 프로젝트 셋업
Phase 1: 법인·테넌트 관리 (테넌트 기반 인프라)
Phase 2: AuditLog & Hash Chain
Phase 3: GHG 계산 엔진
...
```

**판단: ❌ 부적용**

**이유**: Phase 1이 Phase 2의 **직접 전제조건**이다.

| 의존성 | 근거 |
|---|---|
| `audit_logs.tenant_id` | Phase 1에서 `tenants` 테이블과 RLS 정책 생성 후에만 tenant_id 참조 가능 |
| Hash Chain의 `tenantId` | 테넌트별 독립 Hash Chain — 테넌트 엔티티 없으면 체인 구성 불가 |
| RLS 정책 | Phase 1에서 설정한 `app.current_tenant_id` 세션 변수 없으면 audit_logs RLS 동작 안 함 |
| Phase 0에서 스캐폴딩 | ModularityTest, Flyway V1 등 공통 인프라가 Phase 0에서 구성 |

나머지 순서(Phase 3 배출계수, Phase 5 Scope 3, Phase 8 VW)는 현재 plan.md와 일치하므로 Gemini 제안이 사실상 이미 반영됨.

**조치**: 없음. Phase 1 → Phase 2 순서 유지.

## 5. Gemini 리뷰 전반 평가

- **정확도**: ChatGPT 대비 높음. 프로젝트 구조와 아키텍처를 정확히 이해한 상태에서 리뷰
- **사실 오류**: 1건 (G-4 — F-03-08 이미 MVP에 있다는 사실 미인지)
- **설계 이해 오류**: 1건 (G-5 — Phase 1→2 의존성 구조 간과)
- **실질적 개선 제안**: G-2(데이터 이관)는 실제 gap. G-3(DR 절차)은 유의미한 보강
- **긍정 평가의 정확성**: 장점 4건 모두 정확. 아키텍처의 강점을 올바르게 인식
