## 1. 개요 (Executive Summary)

본 보고서는 `esg-t2` 프로젝트의 기획 및 설계 문서를 상세히 검토한 결과입니다. `esg-t2`는 1세대 MVP(`esg-t1`)의 교훈을 바탕으로, 2026년 확정된 **KSSB 1/2(한국판 IFRS S1/S2)** 규제를 완벽히 준수하며 **다법인 연결**, **Scope 3**, **외부 검증 워크스페이스**를 포함하는 엔터프라이즈급 ESG 공시 플랫폼 구축을 목표로 하고 있습니다.

검토 결과, 프로젝트는 매우 높은 수준의 아키텍처 설계와 규제 전문성을 갖추고 있으며, 특히 데이터 무결성(AuditLog Hash Chain)과 테넌트 격리(PostgreSQL RLS) 등 보안과 신뢰성에 강력한 우선순위를 두고 있습니다.

---

## 2. 주요 분야별 검토 결과

### 2.1 아키텍처 및 기술 스택
- **Spring Modulith & Java 25**: 최신 기술 스택(Java 25 Virtual Threads, Spring Boot 4.0)을 활용하여 성능과 유지보수성을 동시에 확보했습니다. 특히 모듈 간 경계를 코드로 강제하는 Modulith 접근 방식은 대규모 시스템 확장에 유리합니다.
- **Domain ≠ Entity 분리**: 영속성 계층과 비즈니스 로직을 엄격히 분리하여, 규제 변경에 따른 도메인 로직 수정이 인프라에 미치는 영향을 최소화했습니다.
- **PostgreSQL RLS (Row-Level Security)**: 애플리케이션 레벨이 아닌 DB 레벨에서 테넌트 격리를 강제하여, 멀티테넌트 환경에서의 데이터 유출 리스크를 원천 차단했습니다.

### 2.2 규제 대응 및 기능성
- **KSSB 1/2 준수**: 2026년 2월 확정된 최신 규제를 반영하고 있으며, 특히 의무 공시가 예상되는 Scope 1/2/3를 MVP 범위에 포함시킨 점이 탁월합니다.
- **데이터 무결성(AuditLog & Hash Chain)**: 공시 데이터의 신뢰성을 위해 SHA-256 해시 체인을 도입한 점은 외부 감사 및 검증 대응 시 강력한 차별점이 될 것입니다.
- **연결 공시 엔진**: 다법인 지분율에 따른 자동 집계와 이중 계상 제거 로직은 실제 상장사들이 겪는 가장 큰 페인 포인트를 정확히 타격하고 있습니다.

### 2.3 구현 계획 및 프로세스
- **TDD 기반 점진적 구축**: 모든 Phase에서 TDD를 강제하고 DoD(Definition of Done)를 명확히 정의하여 코드 품질을 담보하고 있습니다.
- **esg-t1 교훈 반영**: 이전 프로젝트에서 발생한 버그(L-0-01 ~ L-0-16)를 각 Phase의 예방 체크리스트로 포함시킨 점은 매우 모범적인 개발 문화입니다.

## 3. 장점 및 단점 분석 (Pros & Cons)

### 👍 장점 (Pros)
1.  **규제 전문성**: 단순히 수치를 계산하는 것을 넘어, KSSB와 GHG Protocol의 미묘한 요건(95% 임계값, 지출 기반 등)을 시스템적으로 녹여냈습니다.
2.  **신뢰할 수 있는 데이터 추적성**: '수치 → 산식 → 활동 데이터 → 증빙 파일'로 이어지는 역추적(Traceability) 기능은 규제 대응 시스템의 핵심 가치입니다.
3.  **견고한 보안 모델**: RLS와 AuditLog AOP는 엔터프라이즈 환경에서 필수적인 '책임 추적성'과 '데이터 격리'를 완벽하게 지원합니다.
4.  **유연한 산식 엔진(Formula DSL)**: 코드가 아닌 YAML로 산식을 관리하여, 규제 변경 시 재배포 없이 대응 가능한 구조를 갖췄습니다.

### 👎 잠재적 위험 및 단점 (Cons/Risks)
1.  **기술적 복잡도**: Java 25, Spring Modulith, RLS, Hash Chain 등 고급 기술이 대거 도입되어, 초기 온보딩 및 운영 난이도가 높을 수 있습니다.
2.  **성능 병목 가능성**: 대규모 다법인 연결 집계 및 해시 체인 검증 시 DB 부하가 집중될 수 있으므로, Phase 12의 성능 최적화 단계가 매우 중요합니다.
3.  **UI/UX 개발 게이트**: 'UI Phase 시작 전 사용자 승인' 원칙은 좋으나, 복잡한 데이터를 다루는 시스템 특성상 백엔드 설계 단계부터 프론트엔드 프로토타입이 병행되지 않으면 데이터 모델과 UI 간 괴리가 발생할 위험이 있습니다.

## 4. 개선점 및 수정 방향 제안

1.  **UI/UX 프로토타이핑 조기 도입**: 
    - **방향**: Phase 9까지 기다리지 말고, Phase 3(GHG 계산) 단계부터 핵심 화면(데이터 입력 폼, 대시보드)의 와이어프레임을 확정하여 백엔드 API 설계와 동기화할 것을 권장합니다.
2.  **데이터 마이그레이션 전략 구체화**:
    - **개선**: 기존 엑셀이나 타 시스템에서 관리하던 대량의 과거 데이터를 이관하는 `Data Migration Tool` 또는 `Batch Job`에 대한 상세 스펙을 추가할 필요가 있습니다.
3.  **재난 복구(DR) 시나리오 강화**:
    - **개선**: `docs/plan.md`에 백업은 언급되어 있으나, Hash Chain 무결성 오류 발생 시의 구체적인 '데이터 복원 절차(Recovery Playbook)'를 ADR 수준에서 정리하면 더욱 완벽할 것입니다.
4.  **AI 보조 기능의 조기 검토**:
    - **개선**: M+1로 미뤄진 'AI 데이터 검증 보조' 기능을 Phase 6(데이터 수집)의 일부로 포함시켜, 이상값 탐지를 실시간으로 지원하면 사용자 경험이 크게 향상될 것입니다.

## 6. 결론

`esg-t2` 프로젝트 문서는 실제 비즈니스 환경과 규제를 깊이 있게 통찰한 결과물입니다. 특히 **'불변성(Immutability)'**과 **'재현성(Reproducibility)'**을 아키텍처의 핵심 원칙으로 세운 점은 공시 지원 시스템으로서 가장 올바른 방향입니다. 제안드린 UI 프로토타이핑 조기 도입과 데이터 이관 전략만 보강된다면, 시장에서 가장 신뢰받는 ESG 플랫폼으로 자리매김할 수 있을 것으로 확신합니다.
